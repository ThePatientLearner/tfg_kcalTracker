1 · Introducción y arquitectura general

Qué decir (≈45 s): "La aplicación es un tracker de calorías en Android nativo, escrito en Java. El usuario registra lo que come a lo largo del día, busca alimentos desde una base de datos compartida y la app le calcula kcal y macronutrientes sobre su objetivo personalizado."

Capas del proyecto

  • UI Activities: LogInActivity, RegisterActivity, MainActivity, PerfilActivity, ObjetivoActivity, EstadisticasActivity, LogActivity.
  • Modelo models/: UserProfile, FoodProduct, FoodEntry, DailyMacros.
  • Local database/: LocalDatabaseHelper (SQLite).
  • Remoto Firebase: Auth + Realtime Database (/users, /foods).
  • Adapters adapters/: FoodEntryAdapter, FoodSuggestionAdapter, StatLogAdapter.
Idea fuerza: el diseño es offline-first. Primero pinto desde SQLite (instantáneo, sin conexión); Firebase actualiza por encima cuando llega la respuesta. El usuario nunca ve una pantalla en blanco.

private void doLogin(String email, String password) {
    auth.signInWithEmailAndPassword(email, password)
        .addOnCompleteListener(this, task -> {
            if (task.isSuccessful()) {
                user = auth.getCurrentUser();
                updateUi(user);   // → MainActivity
            } else {
                Toast.makeText(this, "Error al iniciar sesión: " + ..., Toast.LENGTH_SHORT).show();
            }
        });
}

Google Sign-In + puente a Firebase

private void firebaseAuthWithGoogle(String idToken) {
    AuthCredential credential = GoogleAuthProvider.getCredential(idToken, null);
    auth.signInWithCredential(credential)
        .addOnCompleteListener(this, task -> {
            if (task.isSuccessful()) checkProfileAndContinue(auth.getCurrentUser());
        });
}
Matiz a comentar: checkProfileAndContinue() consulta /users/{uid} en Firebase. Si existe, voy directo a MainActivity; si no, muestro el popup de perfil para que rellene sus datos. Esto evita pedir dos veces lo mismo a usuarios que ya se habían registrado.

5 · Búsqueda de alimentos en tiempo real

Qué decir: "La búsqueda no golpea Firebase a cada tecla. Uso un debounce de 400 ms con Handler.postDelayed, y normalizo el texto (minúsculas sin acentos) contra el campo nombreLower que grabé en cada alimento."

Debounce con Handler

binding.txtSearchMainActivity.addTextChangedListener(new TextWatcher() {
    @Override public void afterTextChanged(Editable s) {
        String query = s.toString().trim();
        if (searchRunnable != null) searchHandler.removeCallbacks(searchRunnable);
        if (query.length() < 3) { hideSuggestions(); return; }

        searchRunnable = () -> searchFood(query);         // se ejecuta tras 400 ms
        searchHandler.postDelayed(searchRunnable, SEARCH_DELAY_MS);
    }
});

Normalización + filtrado en cliente

String queryNorm = Normalizer.normalize(query.toLowerCase(), Normalizer.Form.NFD)
        .replaceAll("\\p{InCombiningDiacriticalMarks}+", "");

FirebaseDatabase.getInstance(...).getReference("foods")
    .addListenerForSingleValueEvent(new ValueEventListener() {
        @Override public void onDataChange(DataSnapshot snap) {
            for (DataSnapshot child : snap.getChildren()) {
                FoodProduct p = child.getValue(FoodProduct.class);
                if (p.hasValidData() && p.getNombreLower().contains(queryNorm))
                    products.add(p);
                if (products.size() >= 5) break;
            }
            suggestionAdapter.setSuggestions(products);
        }
    });
Limitación honesta: Firebase Realtime DB no soporta contains() nativo, así que descargo /foods y filtro en cliente. Cuando la base crezca mucho se migraría a Firestore con búsqueda por prefijo o a Algolia.

6 · Persistencia local — LocalDatabaseHelper

app/src/main/java/…/database/LocalDatabaseHelper.java
Qué decir: "Extiende SQLiteOpenHelper. Defino tres tablas: perfil de usuario, entradas de comida diarias y totales del día. Todo usa INSERT OR REPLACE, así cada guardado es idempotente."

Las tres tablas

  • user_profile — PK uid (1 fila por usuario).
  • food_entries — PK firebase_key (la misma clave push que en Firebase: así evito duplicados cuando sincronizo).
  • daily_totals — PK compuesta (uid, date), un registro por día.

Guardar una entrada

public void saveFoodEntry(String uid, String date, FoodEntry entry) {
    if (entry.getFirebaseKey() == null) return;
    ContentValues v = new ContentValues();
    v.put(COL_FIREBASE_KEY, entry.getFirebaseKey());
    v.put(COL_ENTRY_UID,    uid);
    v.put(COL_DATE,         date);
    v.put(COL_ENTRY_NAME,   entry.getName());
    v.put(COL_KCAL_PER_100G, entry.getKcalPer100g());
    ...
    db.insertWithOnConflict(TABLE_FOOD_ENTRIES, null, v,
                            SQLiteDatabase.CONFLICT_REPLACE);
}

Migraciones progresivas

@Override public void onUpgrade(SQLiteDatabase db, int old, int newV) {
    if (old < 2) db.execSQL(CREATE_TABLE_FOOD_ENTRIES);
    if (old < 4) db.execSQL("ALTER TABLE ... ADD COLUMN sex ...");
    if (old < 6) db.execSQL("ALTER TABLE ... ADD COLUMN carbs_per_100g ...");
    if (old < 7) db.execSQL("ALTER TABLE ... ADD COLUMN total_fat ...");
}
Detalle técnico interesante: si actualizo la app en una versión posterior no borro la tabla (DROP), sino que añado columnas. Así los usuarios que ya tenían la app no pierden su historial.

if (localProfile != null) updateProfileUI(localProfile);

// 2) Firebase refresca por encima
refUsers.child(user.getUid()).addListenerForSingleValueEvent(...);

Refactor clave: separar "leer" de "persistir"

Qué decir: "Durante el desarrollo detecté un bug: al cargar un día sin conexión, la app sobrescribía Firebase con totales a 0 mientras SQLite aún estaba pintando. La solución fue partir la función en dos, y que los flujos de carga nunca escribieran en Firebase."
Punto Antes Ahora
Logout signOut() + sp.clear() solo signOut() (conserva last_reset_date)
updateDailyDisplay() Actualiza UI + sync Firebase Solo actualiza UI (función pura de lectura)
addFoodEntry() updateDailyDisplay() persistAndUpdateDailyDisplay() ✓ persiste
"Cambiar gramos" updateDailyDisplay() persistAndUpdateDailyDisplay() ✓ persiste
"Eliminar" updateDailyDisplay() persistAndUpdateDailyDisplay() ✓ persiste
loadTodayFoodEntries Limpia local y adapter siempre Solo limpia si Firebase tiene al menos una entrada (si responde vacío, preserva lo local)
Resto de cargas (loadTodayFoodEntriesFromLocal, updateProfileUI, changeSelectedDate…) updateDailyDisplay() escribía a Firebase updateDailyDisplay() ya no toca Firebase — solo pinta 
// UI pura, sin efectos laterales
private void updateDailyDisplay() { ... // solo repinta barras y textos }

// Única ruta que persiste: se llama cuando el usuario ACTÚA
private void persistAndUpdateDailyDisplay() {
    updateDailyDisplay();
    syncDailyTotals(adapter.getTotalKcal(), adapter.getTotalCarbs(), ...);
}
Regla resultante: solo se escribe en Firebase/SQLite cuando el usuario añade, edita o elimina una comida. Las rutas de carga son de solo lectura. Esto evita la condición de carrera "local pinta antes, remoto llega vacío y machaca los totales con 0".

8 · Cálculo IMC, TMB y objetivo de kcal

Qué decir: "Cuando cargo el perfil, calculo el IMC para clasificar al usuario (bajo peso, normal, sobrepeso, obesidad I/II/III) y estimo su gasto energético diario (TDEE) con la fórmula de Mifflin-St Jeor. Según la categoría, ajusto el objetivo de kcal."
double heightM = profile.getHeight() / 100.0;
double imc     = profile.getWeight() / (heightM * heightM);
double tmbAdj  = "Mujer".equals(profile.getSex()) ? -161 : 5;
double tmb     = (10 * weight) + (6.25 * height) - (5 * age) + tmbAdj;
double tdee    = tmb * 1.375;   // actividad ligera

if (imc < 18.5)      kcalObjetivo = tdee * 1.15;  // bajo peso → superávit suave
else if (imc < 25)   kcalObjetivo = tdee;         // normal → mantenimiento
else if (imc < 30)   kcalObjetivo = tdee * 0.85;  // sobrepeso → déficit 15 %
else                kcalObjetivo = tdee * 0.75;  // obesidad → déficit 25 %

// Suelo de seguridad: nunca por debajo de 1200 (mujer) / 1500 (hombre)
double minSafe = "Mujer".equals(profile.getSex()) ? 1200 : 1500;
if (kcalObjetivo < minSafe) kcalObjetivo = minSafe;

// Distribución estándar de macros
carbsObjetivo   = (kcalObjetivo * 0.50) / 4.0;   // 50 % carbos (4 kcal/g)
proteinObjetivo = (kcalObjetivo * 0.20) / 4.0;   // 20 % proteína (4 kcal/g)
fatObjetivo     = (kcalObjetivo * 0.30) / 9.0;   // 30 % grasa  (9 kcal/g)
Por qué es relevante: es la única lógica "de negocio" de la app y da personalización real. Los valores nunca bajan de un suelo de seguridad para no recomendar dietas peligrosas.

10 · Cierre — puntos fuertes y mejoras

Puntos fuertes del código

  • Offline-first consistente: todas las pantallas pintan primero desde SQLite.
  • Sincronización con la misma clave (firebase_key) en local y remoto → sin duplicados.
  • Debounce de búsqueda para no saturar Firebase.
  • Migraciones progresivas de SQLite que preservan datos del usuario.
  • Separación clara UI / modelo / persistencia / adaptadores.

Mejoras futuras (para la pregunta del tribunal)

  • Migrar a Firestore y búsqueda por prefijo o Algolia para escalar el catálogo.
  • Fusionar LocalDatabaseHelper con Room u ORMLite (anotaciones en vez de SQL crudo).
  • Patrón MVVM con ViewModel + LiveData para desacoplar la Activity.
  • Tests instrumentados de los cálculos nutricionales.
Frase de cierre sugerida: "En resumen: la app aplica un patrón offline-first con Firebase como fuente remota y SQLite como caché local, coordinados por una misma clave. La lógica nutricional queda aislada y es fácil de sustituir por una más avanzada si se quisiera ampliar el proyecto."
